<div class="auto-hide-last-sibling-br paragraph-JOTKXA paragraph-element br-paragraph-space">在当今数字化时代，Java 应用程序的安全性至关重要。代码审计作为保障应用安全的关键环节，能够帮助我们提前发现潜在的安全漏洞。本文将详细介绍 Java 代码审计的思路方法以及常见基础漏洞的审计要点，为安全从业者和开发人员提供参考。</div>
<h2 class="header-vfC6AV auto-hide-last-sibling-br">一、代码审计思路和方法</h2>
<div class="auto-hide-last-sibling-br paragraph-JOTKXA paragraph-element br-paragraph-space">进行 Java 代码审计时，可采用以下四种主要思路和方法，它们各有优劣，适用于不同场景：</div>
<ol class="auto-hide-last-sibling-br">
 	<li>
<div class="auto-hide-last-sibling-br paragraph-JOTKXA paragraph-element br-paragraph-space"><strong>逆向追踪（回溯变量）</strong></div>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>方法</strong>：从敏感函数的参数入手，回溯变量的来源和传递过程，判断其是否可控且未经过严格过滤。例如，可依据 SQL 语句的特征来分析是否存在注入漏洞。</li>
 	<li><strong>优点</strong>：通过搜索敏感关键字，能快速、高效地挖掘到定向漏洞。</li>
 	<li><strong>缺点</strong>：由于未通读代码，对程序整体框架了解不深入，可能遗漏逻辑漏洞。</li>
</ul>
</li>
 	<li>
<div class="auto-hide-last-sibling-br paragraph-JOTKXA paragraph-element br-paragraph-space"><strong>正向追踪（跟踪变量）</strong></div>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>方法</strong>：先找出接收外部参数的文件，然后跟踪变量的传递路径，观察其是否传入高危函数或存在代码逻辑漏洞。</li>
 	<li><strong>优点</strong>：挖掘范围比逆向追踪更全面。</li>
 	<li><strong>缺点</strong>：效率可能不如逆向追踪。</li>
</ul>
</li>
 	<li>
<div class="auto-hide-last-sibling-br paragraph-JOTKXA paragraph-element br-paragraph-space"><strong>直接挖掘功能点漏洞</strong></div>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>方法</strong>：根据经验判断应用中哪些功能点可能存在漏洞，然后直接阅读该部分代码。</li>
</ul>
</li>
 	<li>
<div class="auto-hide-last-sibling-br paragraph-JOTKXA paragraph-element br-paragraph-space"><strong>通读全文</strong></div>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>方法</strong>：通读整个应用程序的代码，以了解其业务逻辑和整体架构。建议先从程序入口文件（如 <code>index</code> 文件）、函数集文件（如 <code>functions</code>、<code>common</code>）、配置文件（如 <code>config</code>）和安全过滤文件（如 <code>filter</code>、<code>safe</code>、<code>check</code>）入手。</li>
 	<li><strong>优点</strong>：能更好地了解程序架构和业务逻辑，挖掘出更多有价值的逻辑漏洞。</li>
 	<li><strong>缺点</strong>：耗时较多，对于大型程序来说工作量较大。</li>
</ul>
</li>
</ol>
<h2 class="header-vfC6AV auto-hide-last-sibling-br">二、基础漏洞审计</h2>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">1. SQL 注入</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>漏洞原理</strong>：程序未对用户输入进行正确检查，直接以拼接方式将输入带入 SQL 语句中，导致攻击者可构造恶意 SQL 语句影响数据库操作。</li>
 	<li><strong>SQL 语句执行方式</strong>：
<ul class="auto-hide-last-sibling-br">
 	<li><strong>JDBC</strong>：通过 <code>java.sql.Statement</code> 对象进行直接拼接，容易产生注入；而 <code>java.sql.PreparedStatement</code> 使用预编译，相对安全。</li>
 	<li><strong>MyBatis</strong>：使用 <code>$</code> 符号进行拼接会造成注入，而 <code>#</code> 符号使用预编译，基本安全。</li>
 	<li><strong>Hibernate</strong> 也可能存在相关注入风险，需具体分析。</li>
</ul>
</li>
 	<li><strong>审计关键词</strong>：<code>Statement</code>、<code>createStatement</code>、<code>PrepareStatement</code>、<code>select</code>、<code>update</code>、<code>insert</code>。</li>
 	<li><strong>常见注入类型</strong>：
<ul class="auto-hide-last-sibling-br">
 	<li>参数动态拼接。</li>
 	<li>预编译错误。</li>
 	<li><code>ORDER BY</code> 注入：<code>ORDER BY</code> 无法预编译，需要使用拼接方式，易出现漏洞。</li>
 	<li><code>LIKE</code> 模糊查询注入。</li>
 	<li><code>IN</code> 查询注入：在 <code>IN</code> 语句中使用 <code>#</code> 会导致整个参数作为一个整体拼接进去，容易出现注入。</li>
</ul>
</li>
 	<li><strong>审计方法</strong>：通过关键词搜索，找到关键字后查看周围是否安全调用外部变量或是否有过滤。若没有，则追踪调用栈查找引用位置。</li>
 	<li><strong>二次注入</strong>：在注册、登录等位置注入包含特殊字符的 SQL 语句，在后续查询用户信息的过程中可能导致二次注入。</li>
</ul>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">2. 文件上传</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>漏洞原理</strong>：程序员对文件上传的控制不足或处理有缺陷，导致用户可以上传可执行的动态脚本文件。问题的关键在于服务器如何处理和解释上传的文件。</li>
 	<li><strong>审计关键词</strong>：<code>org.apache.commons.fileupload</code>、<code>java.io.File</code>、<code>MultipartFile</code>、<code>RequestMethod</code>。</li>
 	<li><strong>审计方法</strong>：
<ul class="auto-hide-last-sibling-br">
 	<li>关注上传表单代码段。</li>
 	<li>关注文件上传后的后缀校验、文件类型黑白名单、Content-Type 黑白名单，以及（如果为图片）图片库检测。</li>
 	<li>检查是否存在以下常见问题：未做任何过滤、仅在前端进行过滤、只判断 <code>Content-Type</code>、后缀过滤不全、读取后缀方式错误（例如，不安全的后缀获取方式为 <code>fileName.indexOf(&quot;.&quot;)</code>）。</li>
 	<li>后端需要对文件类型、大小、文件名和内容进行验证。</li>
</ul>
</li>
</ul>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">3. XSS (跨站脚本攻击)</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>漏洞原理</strong>：通过注入恶意代码（通常是 JavaScript）到网页中，当用户访问该页面时，恶意代码被执行，可能导致用户信息泄露等问题。</li>
 	<li><strong>漏洞分类</strong>：
<ul class="auto-hide-last-sibling-br">
 	<li><strong>反射型 XSS</strong>：非持久化，注入的恶意代码通过服务器的响应返回到前端页面并被执行。</li>
 	<li><strong>存储型 XSS</strong>：持久化，注入的恶意代码存储在服务器中，用户访问页面时触发。</li>
 	<li><strong>DOM 型 XSS</strong>：通过修改页面的 DOM 节点触发。</li>
</ul>
</li>
 	<li><strong>审计关键词</strong>：<code>request.getParameter</code>、<code>response.getWriter().print()</code>、<code>response.getWriter().writer()</code>、<code>&lt;%=</code>、<code>${</code>、<code>&lt;c:out</code>、<code>&lt;c:if</code>、<code>&lt;c:forEach</code>、<code>ModelAndView</code>、<code>ModelMap</code>。</li>
 	<li><strong>修复手段</strong>：采用全局过滤器、安全应用程序接口（如 ESAPI / 谷歌的 xssProtect）、HTML 实体编码等方式进行防御。</li>
</ul>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">4. 目录穿越</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>漏洞原理</strong>：没有校验文件名中是否包含 <code>../</code> 等特殊字符及对用户访问文件进行限制，导致攻击者可访问未授权的目录和文件。</li>
 	<li><strong>审计关键词</strong>：<code>Path</code>、<code>java.io.File</code>、<code>FileInputStream</code>、<code>file.read</code>。</li>
 	<li><strong>修复手段</strong>：进行文件名过滤，检测路径是否合法。</li>
</ul>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">5. URL 重定向</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>审计关键词</strong>：<code>return</code>、<code>redirect</code>、<code>url</code>、<code>jump</code>、<code>goto</code>、<code>target</code>、<code>link</code>、<code>new ModelAndView</code>、<code>redirectAttribute</code>、<code>response.setHeader</code>。</li>
 	<li><strong>修复手段</strong>：控制跳转域名，将跳转值写为固定值；设置只能根据路径跳转。</li>
</ul>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">6. 命令执行</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>漏洞原理</strong>：代码未对用户可控参数做过滤，导致恶意命令被拼接到正常命令中执行，可能对服务器造成严重破坏。</li>
 	<li><strong>执行命令方法</strong>：
<ul class="auto-hide-last-sibling-br">
 	<li><code>java.lang.ProcessBuilder.start()</code>。</li>
 	<li><code>java.lang.Runtime.getRuntime.exec()</code>：其 String 型重载在执行时会使用 <code>StringTokenizer</code> 函数按空格、制表符等分割命令，可能导致无法执行；而数组型重载可以直接调用 <code>ProcessBuilder</code> 执行，相对安全。</li>
</ul>
</li>
 	<li><strong>审计关键词</strong>：<code>getParameter</code>、<code>ProcessBuilder</code>、<code>start()</code>、<code>Runtime.getRuntime.exec()</code>。</li>
 	<li><strong>修复手段</strong>：避免直接使用 <code>getRuntime</code>，并过滤输入的恶意字符。</li>
</ul>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">7. XXE (XML 外部实体注入)</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>漏洞原理</strong>：应用程序在解析 XML 输入时，未禁止外部实体的加载，导致攻击者可加载恶意外部文件，获取敏感信息等。</li>
 	<li><strong>审计关键词</strong>：<code>XMLReader</code>、<code>SAXBuilder</code>、<code>SAXReader</code>、<code>SAXParserFactory</code>、<code>Digester</code>、<code>DocumentBuilderFactory</code>。</li>
 	<li><strong>审计方法</strong>：
<ul class="auto-hide-last-sibling-br">
 	<li>检查 XML 解释器的配置，看是否禁用了 XML 外部实体。</li>
 	<li>查看代码中是否使用了上述关键词中列出的 XML 解析接口或类。</li>
</ul>
</li>
 	<li><strong>修复手段</strong>：禁用 DTD / 禁用外部实体。</li>
</ul>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">8. SSRF (服务端请求伪造)</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>漏洞原理</strong>：攻击者伪造服务器发出请求，向内部网络或其他受保护资源发起请求，而服务器未进行过滤和限制，可能导致内部信息泄露等问题。</li>
 	<li><strong>审计关键词</strong>：<code>HttpURLConnection.getInputStream</code>、<code>URLConnection.getInputStream</code>、<code>HttpClient.execute</code>、<code>URL.openStream</code>、<code>ImageIO.read</code>。</li>
 	<li><strong>审计方法</strong>：
<ul class="auto-hide-last-sibling-br">
 	<li>快速找到 HTTP 请求操作函数。</li>
 	<li>测试是否可以访问内部地址（如 <code>http://127.0.0.1</code> 或内网 IP）或使用不同的 URL 协议（如 <code>file</code> 协议）。</li>
</ul>
</li>
 	<li><strong>修复手段</strong>：
<ul class="auto-hide-last-sibling-br">
 	<li>控制跳转域名，最好在源码中写为固定值。</li>
 	<li>设置只能根据路径跳转。</li>
</ul>
</li>
</ul>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">9. 反序列化</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>漏洞原理</strong>：应用程序在反序列化过程中，未对恶意数据进行验证和过滤，导致攻击者可以劫持字节流，植入恶意代码并被执行。</li>
 	<li><strong>必要条件</strong>：存在利用链（exploit chain）和触发点（trigger point）。</li>
 	<li><strong>审计关键词</strong>：<code>ObjectInputStream.readObject</code>、<code>XMLDecoder.readObject</code>、<code>yaml.load</code>、<code>XStream.fromXML</code>、<code>ObjectMapper.readValue</code>、<code>JSON.parseObject</code>。</li>
 	<li><strong>审计方法</strong>：
<ul class="auto-hide-last-sibling-br">
 	<li>检查代码中是否使用了上述关键词。</li>
 	<li>关注流行的 Java 框架（如 Apache Struts、Apache Commons Collections、Jackson）中存在的反序列化漏洞。</li>
</ul>
</li>
</ul>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">10. CSRF (跨站请求伪造)</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>漏洞原理</strong>：攻击者盗用用户的身份，以用户的名义发起恶意请求，而服务器没有进行身份识别，可能导致用户数据被篡改等问题。</li>
 	<li><strong>审计方法</strong>：检查是否对 <code>Referer</code>、<code>Token</code> 等参数进行了严格的检测。</li>
</ul>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">11. 逻辑漏洞</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>漏洞原理</strong>：在软件设计过程中存在错误或疏忽，导致未正确处理输入、验证用户身份、授权访问等，从而引发安全风险。</li>
 	<li><strong>审计方法</strong>：逻辑漏洞的产生不在于代码本身，而在于业务逻辑的缺陷，因此需要深入理解业务逻辑，才能进行测试。</li>
 	<li><strong>常见漏洞类型</strong>：
<ul class="auto-hide-last-sibling-br">
 	<li><strong>验证码漏洞</strong>：验证码爆破、回显、未与手机绑定、修改返回包绕过、转发、固定验证码等。</li>
 	<li><strong>支付漏洞</strong>：任意金额修改、负数购买、越权支付、修改运费、优惠券修改、四舍五入漏洞、整数溢出等。</li>
 	<li><strong>并发漏洞（条件竞争）</strong>：后台未对操作加锁，导致并发操作产生安全问题，例如点赞或领取优惠券等。</li>
 	<li><strong>越权漏洞</strong>：攻击者修改用户身份识别参数（如 <code>id</code>、<code>uid</code>、<code>name</code> 等），进行不属于自己的操作，例如删除评论、查看他人信息等。</li>
</ul>
</li>
</ul>
<h3 class="header-vfC6AV auto-hide-last-sibling-br">12. SpEL 表达式注入</h3>
<ul class="auto-hide-last-sibling-br">
 	<li><strong>漏洞原理</strong>：程序未对用户输入进行验证，直接通过解析引擎解析 SpEL 表达式，可能导致恶意代码执行。</li>
 	<li><strong>审计关键词</strong>：<code>SpelExpressionParser</code>、<code>parseExpression</code>、<code>expression.getValue()</code>。</li>
 	<li><strong>修复手段</strong>：使用 <code>SimpleEvaluationContext</code>。</li>
</ul>
 
<div class="auto-hide-last-sibling-br paragraph-JOTKXA paragraph-element br-paragraph-space">通过以上对 Java 代码审计思路方法和常见漏洞的梳理，希望能帮助大家在实际工作中更好地进行代码审计，提升 Java 应用程序的安全性。在实际审计过程中，还需结合具体业务场景和代码细节，进行全面、深入的分析。</div>
